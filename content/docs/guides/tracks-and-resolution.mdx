This guide explains the difference between resolved and unresolved tracks, and how to safely narrow their types using type guards.

## Type Guards

Use `isResolved()` and `isUnresolved()` to narrow track types safely.

```ts twoslash copy wrap title="track-type-guards.ts"
import { isResolved, isUnresolved } from 'hoshimi';
import type { Track, UnresolvedTrack } from 'hoshimi';

declare const input: Track | UnresolvedTrack;

if (isResolved(input)) {
  console.log(input.info.identifier);
}

if (isUnresolved(input)) {
  console.log(input.info.title);
}
```

## Queue Operations with Mixed Tracks

Add both resolved and unresolved tracks to the queue—the system will resolve them as needed.

```ts twoslash copy wrap title="queue-mixed-tracks.ts"
import type { Player, Track, UnresolvedTrack } from 'hoshimi';

declare const player: Player;
declare const items: Array<Track | UnresolvedTrack>;

for (const item of items) {
  await player.queue.add(item);
}

if (!player.isPlaying()) {
  await player.play();
}
```

## Key Concepts

- **Resolved tracks**: Have a Lavalink identifier and metadata (duration, artist, etc.).
- **Unresolved tracks**: Only have query info (title, author); need node resolution.
- **Type guards**: Always use `isResolved()` before accessing identifier-dependent properties.
- **Queue flexibility**: The queue accepts both types; unresolved tracks are resolved during playback.
